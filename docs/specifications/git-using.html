首先说明几个必要的分支的用途：

  • master：整个源代码。其实，它是一类分支的代表，即源代码打包成形的分支；利用该类分支可以编译出一个可用软件
  • util：主要放stringutil啦，timeutil啦。就是把大家认为在很多地方都会用到的公共代码放到这里。这个过程是个实践的过程。这类分支的特点是按模块划分，只不过util是个所有其它分支都会用到的

PS：没错，就是增量的概念。但是由于merge是要认为从分支A加到分支B的手动过程，分支B的内容不会自动和分支A的内容同步。所以，请严格按照模块和源代码打包的思想去建立分支。

接下来说一下，我们将来开发的日常：

  • 打开Eclipse，Fetch你正在开发的分支，以及你所直接merge过的分支。这里举个例子，假如我开发一个模块C，我跟和其他开发模块C的童鞋，先Fetch我的分支C；接下来，假如模块C中代码调用了模块D和模块E中的代码，那么我再Fetch分支D和分支E的代码，然后merge到我的分支C中来
  • Switch To你自己的branch，这个其实不要理会的，你开发哪个模块就切换到哪个分支就好了。记得遵守日常1先！
  • 开始你的开发吧。。。Commit的时候分清楚自己在哪个分支，看仔细了，否则还得删删改改。

PS：我们常看到我们自己的模块调用的那个模块中的代码不够好，怎么办？不要在你的分支中改它，在Issues里@相应的人；如果实在手痒，改就改了，反正第二天还要merge回去，所以。。。So，原则就是，不要把大的分支merge到小的分支中去！

最后来说一下，第一次clone时的几点：

  • 选择要clone的分支时，只选择你要开发的模块的分支和其直接依赖的模块的分支就好。这个东西的说明，也在文档里面找
  • 如果要学习整个源代码，勾上master没错
  • 把源代码从服务器拉到本地后，要是建立在此基础上开发新模块，就new一个分支，source选择你依赖的一个模块的分支，其它依赖的模块再一个个merge进来
